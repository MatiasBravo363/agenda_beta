# Iteración: 1.0.21 — Audit fuentes de requests + endurecimiento
Fecha: 2026-04-30
Estado: esperando revisión

## Requerimiento

Tras la 1.0.20, el reporte de query performance de Supabase reveló que el 97.72% del CPU lo consume `set_config` (query inicial de PostgREST por cada HTTP request). Total: 226M calls acumuladas. La causa real del peak NO son queries lentas — es **volumen de requests HTTP**.

Decisión: ejecutar la **opción C** del análisis previo — auditar en profundidad las fuentes potenciales del lado nuestro y endurecer las identificadas.

## Hallazgos del audit

### Fuentes accionables internas (3)
1. **`publicGuard` llama `getSession()` siempre** (auth.guard.ts:22). No chequea el signal antes.
2. **`/status` sin throttle** (status.component.ts:82-84). Ejecuta query a `schema_version` en cada `ngOnInit`. Ruta pública → potencial abuso por bots.
3. **`status.component.ts` con APP_VERSION stale** (`'1.0.13'` hard-coded vs package.json en 1.0.20).

### Patrones descartados (no son problema)
- **Polling oculto**: no hay `setInterval`, `realtime`, `interval()` de RxJS. Los `setTimeout` que existen son debounces acotados (calendar y dashboard).
- **Service Worker / PWA**: no hay configuración.
- **Sentry tracesSampleRate**: 0 (no genera tráfico).
- **Vercel Analytics**: pega a vercel-scripts.com, no a Supabase.
- **`authGuard`**: chequea signal primero, solo invoca `getSession()` si está null. Patrón correcto, no se toca.

### Áreas externas a investigar (sin código)
- **Vercel Logs** últimas 24h para endpoints más hiteados.
- **Supabase Auth Logs** para sesiones con `TOKEN_REFRESHED` excesivo.
- **`pg_stat_statements_reset()`** para baseline limpia.

## Criterios de aceptación

- [ ] `publicGuard` no llama `getSession()` cuando el usuario ya está autenticado por signal.
- [ ] `/status` recargado dentro de 60s del último check exitoso muestra "cacheado", no hace query.
- [ ] Botón "Volver a chequear" en `/status` sigue forzando re-check (ignora throttle).
- [ ] Sidebar muestra V1.0.21.
- [ ] `/status` muestra `agenda-beta@1.0.21` (no más 1.0.13).

## Pasos manuales (críticos para diagnóstico)

Estos NO son código y los tiene que hacer el usuario para que el análisis post-1.0.21 sea válido:

### Antes de mergear (recomendado)
- En Vercel Dashboard → proyecto agenda-beta → **Logs**, filtrar últimas 24h y exportar/screenshot top 20 endpoints. Esto da baseline.

### Después de mergear y deploy
1. **Reset SQL** en Supabase Dashboard SQL Editor:
   ```sql
   SELECT pg_stat_statements_reset();
   ```
   Esto borra stats acumulados históricos. Empezamos a contar de 0.

2. **Esperar 24h** de uso normal.

3. **Re-exportar el reporte de Query Performance** (mismo CSV que enviaste antes). Comparar:
   - Cantidad de calls a `set_config` (debería bajar significativamente si la 1.0.21 + el reset fueron suficientes).
   - Si sigue alto, indica fuente externa (bot, monitor, abuse). Pasar a paso 4.

4. **Si el peak persiste**: revisar en paralelo:
   - Vercel Logs últimas 24h: buscar IPs/endpoints anómalos.
   - Supabase → Auth → Logs: filtrar `TOKEN_REFRESHED` por usuario.

## Riesgo aceptado

- El throttle de `/status` puede dar un OK cacheado si Supabase cae justo después de un check exitoso. Mitigación: el botón "Volver a chequear" fuerza re-check; el cacheo es de 60s máximo.
- Cambios al `publicGuard` mantienen la misma semántica (mismo resultado final), solo evitan HTTP cuando el signal ya tiene la respuesta. Tests cubren los casos.
