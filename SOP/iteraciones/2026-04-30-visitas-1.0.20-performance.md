# Iteración: 1.0.20 — Performance / scope reduction
Fecha: 2026-04-30
Estado: esperando revisión

## Requerimiento

Supabase reportó CPU > 80% post 1.0.19. Tras inventario de queries, el usuario eligió 4 cambios para mitigar la carga:

1. `/visitas/calendario` default semana actual (filtro por rango visible).
2. `/dashboard` default semana + cap de 6 meses.
3. Eliminar vista `/historial` completamente.
4. `count: 'exact'` → `count: 'estimated'` en `listPaged()` (quick win adicional sumado en este release).

Ahorro estimado total: ~70% de la carga CPU actual.

## Criterios de aceptación

- [ ] `/visitas/calendario` al cargar fetcha solo el rango visible (semana actual por default), no las 500 visitas globales.
- [ ] Navegar prev/next en el calendario dispara nuevo fetch con el nuevo rango (debounce 250ms).
- [ ] `/dashboard` default semana actual (lunes a domingo).
- [ ] Si el usuario selecciona un rango > 6 meses en el dashboard, banner amarillo bloquea charts y ofrece volver a semana actual.
- [ ] Cambiar fecha desde/hasta en dashboard dispara reload server-side via `listPaged`.
- [ ] Sidebar ya no muestra "Historial". Tipear `/historial` cae en catch-all `**` y redirige.
- [ ] Tabla `visitas_historial` y trigger 017 intactos (no se tocan).
- [ ] `listPaged()` usa `count: 'estimated'`. El "Total" del footer puede estar ±5% (aceptable).

## Notas de implementación

### Calendar: hook datesSet de FullCalendar
- `info.startStr` / `info.endStr` vienen como `YYYY-MM-DDTHH:MM:SS-03:00` (tz local). El método `onDatesSet` extrae solo la parte de fecha (`slice(0,10)`) y resta 1 día a `endStr` porque FullCalendar lo pasa como exclusivo.
- Debounce 250ms evita fetch storm cuando el usuario hace click rápido en prev/next.
- `reloadVisitas()` hace 2 queries: una con rango temporal y otra para `en_cola` (sin fecha — el sidebar lateral las muestra). Merge por id sin duplicados.

### Dashboard: rango excedido bloquea reload
- `cargarVisitas()` chequea `rangoExcedido()` antes de fetch. Si excede, setea `items.set([])` y los charts muestran placeholder "Sin datos en el rango filtrado".
- `onFechaChange` con debounce 300ms en los inputs de fecha (evita pegarle al server por cada keystroke al tipear la fecha).

### Eliminación de history
- Solo UI/service. La tabla DB `visitas_historial` y trigger 017 (`log_visita_change`) siguen activos. Si en el futuro se quiere reactivar la vista, los datos están ahí.
- Permiso `historial.ver` queda en `PermisoCodigo` (TS) y `permisos` (DB) sin chequear en ningún lado — limpieza opcional para futuro release.
